## What
Новая top-level идея `spec-flat-tags`: представлять требования плоским списком + энфорсимые теги, «дерево» = проекция `group-by`. Прецедент — conf-модуль PetBox уже на этой схеме.

## Why
Из догфуда `tasks-md-render` по методологии вышли на вопрос структуры spec. Ключевые опоры: (1) TemporalStore по факту плоский, `l1/l2/l3` — форсинг-функция (свободный path агенты игнорировали); (2) «энфорсимый слот заполняют, свободное поле игнорируют»; (3) теги дают сквозные требования + roll-up по любой группировке + глубина = стопка ключей group-by, чего строгое дерево не даёт; (4) пользователь подтвердил: conf-модуль уже сделан по этой схеме, к спеке просто не применяли.

## Args
Тело `spec-flat-tags` (nodeId ce691c37…): плоские строки + энфорсимые теги (`area:*`/`concern:*`), дерево = group-by (проекция, глубина = стопка ключей, перестройка = смена параметра вью), сквозные req без M:N-костылей, roll-up по любой группировке. Энфорсмент несущий: контролируемый словарь / обязательность / авто-классификация на ingest (ср. type=auto, авто-тип дистилляции). Мини-пример R1..R6 с двумя проекциями (area / concern). Связь с методологией: ценность spec сохраняется, иерархия = проекция; открыто — маппинг computed type-aware статуса на per-grouping roll-up.
