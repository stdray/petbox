# Deploy control-plane — наброски

> Статус: ранние наброски, **не идея и не спека**. Место для размышлений перед тем, как (возможно) завести идею в методологии. Канон сюда не переносить, пока не дозреет.

## Контекст и рамка

PetBox дошёл до состояния, когда проект доводится до рабочего за считанные минуты, и узким местом становится **деплой, мониторинг и управление** этими проектами. Все нагрузки — docker-контейнеры, которые на вход получают URL + ключ PetBox + тег-вектор и сами ходят наружу к PetBox (топология «звезда»: сервисы общаются с центром и с внешними API, почти не друг с другом).

Выбранная модель: **PetBox — control plane** (мозг: реестр узлов, желаемое состояние, конфиг, UI, отображение «ключ+вектор → параметры контейнера»). На каждой машине — **тонкий pull-агент** (руки: цикл сверки желаемого с фактическим). Агент только *исходящим* HTTPS опрашивает PetBox — никакого входящего порта (проходит сквозь NAT и сетевые ограничения, безопаснее, делает локальный ПК бесплатной нодой).

Хотелки, которые это закрывает: перемещать сервис по серверам, держать копии на разных серверах, старт/стоп через UI, локальный ПК как нода, запуск с параметрами из ключа+вектора.

Этот документ разбирает **три механизма из «большого» оркестратора** (rolling, canary, service mesh) — что это, зачем, и насколько они нужны *тебе*. Короткий вывод: ни один не нужен сейчас, модель деградирует изящно, а дешёвые версии части из них получаются почти даром из desired-state модели.

## 1. Rolling deploy (постепенное обновление)

### Проблема, которую решает

Наивный деплой в нашей модели — **stop-then-start**: убили старый контейнер, подняли новый с новым образом. Между этими двумя моментами сервис **недоступен** — для HTTP-сервиса это окно 502-х на несколько секунд, пока контейнер стартует и проходит healthcheck.

### Что такое rolling

Обновляем экземпляры **не все разом, а по одному (или батчами)**:

1. Поднять новый экземпляр рядом со старым.
2. Дождаться, пока он станет healthy.
3. Переключить на него трафик, погасить старый.
4. Повторить для следующего экземпляра.

Итог — обновление без (или почти без) простоя. Это дефолт в Kubernetes/Nomad/Swarm.

### Что для этого нужно

- **Больше одного экземпляра** ИЛИ возможность кратко держать старый и новый бок о бок.
- **Reverse-proxy / load-balancer** перед сервисом, который умеет переключать трафик с старого на новый (Caddy / Traefik / nginx).
- **Healthcheck**, по которому понятно «новый готов принимать трафик».

### Насколько это нужно тебе

Почти не нужно, и вот почему:

- **Боты не имеют входящего трафика.** Бот переподключается к Telegram/внешнему API сам после рестарта; «окно простоя» для него невидимо. Stop-then-start полностью устраивает.
- **Для немногих HTTP-сервисов** rolling *дорог именно у тебя*: на 1–2 ГБ держать два экземпляра одного сервиса одновременно может банально не влезть в память. То есть классический rolling упирается в твоё узкое место — RAM.

Дешёвая золотая середина, если для конкретного HTTP-сервиса простой всё же неприемлем: поставить **один общий reverse-proxy** (Caddy/Traefik) на ноду, поднимать новый контейнер на эфемерном порту, после healthcheck переключить proxy и погасить старый. Это «recreate с проверкой здоровья» на одной ноде. Для ботов — не заморачиваться вообще.

**Вывод:** rolling — это про HTTP-сервисы с непрерывным трафиком, которых у тебя мало, и он конфликтует с дефицитом RAM. Закладывать в модель не нужно; при необходимости решается одним shared-proxy, а не оркестратором.

## 2. Canary (релиз на канарейке)

### Проблема, которую решает

Новая версия может быть сломана. Если выкатить её сразу на 100%, баг ловят **все** пользователи/потребители одновременно.

### Что такое canary

Выкатываем новую версию на **маленький срез** — один экземпляр / 5% трафика / одну ноду, — **наблюдаем** (ошибки, латентность, метрики), и только потом:

- если хорошо → промоутим до 100%;
- если плохо → откатываем, пострадал только срез.

«Канарейка в шахте»: маленькая жертва предупреждает о беде раньше, чем накроет всех.

### Что для этого нужно

- **Разделение трафика** (proxy/mesh умеет лить N% на canary), ИЛИ несколько экземпляров и возможность направить на canary часть нагрузки.
- **Хорошая наблюдаемость**, чтобы сравнить canary с базовой версией (error rate, latency).

### Насколько это нужно тебе

Полноценный traffic-split не нужен (нужен proxy/mesh, которого нет; у ботов вообще нет «процента трафика», который можно лить). **Но** дешёвая канарейка ложится на desired-state модель почти бесплатно:

> **Canary by node.** Образ (`image_digest`) хранится per-deployment. Ставишь новый digest на **одну ноду** (удобно — на эфемерный локальный ПК или одну VPS), смотришь её heartbeat / логи / ошибки через PetBox. Хорошо — бампаешь digest у остальных deployment'ов. Плохо — откатываешь digest на одной ноде. Промоут ручной, без proxy, без split.

То есть ты получаешь «канарейку по нодам» даром из того, что digest уже живёт на уровне записи деплоя. Это не процентный canary, но для твоего масштаба — ровно то, что надо.

**Вывод:** настоящий canary с traffic-split не нужен; per-node staging digest даёт 90% пользы за 0 инфраструктуры.

## 3. Service mesh (сервисная сетка)

### Проблема, которую решает

Когда сервисов много и они активно общаются **друг с другом** (east-west трафик), вылезает пачка сквозных проблем на *каждом* межсервисном вызове:

- **Service discovery** — где сейчас сервис B? Он мог переехать на другую ноду.
- **mTLS** — шифровать и аутентифицировать трафик между сервисами.
- **Retries / timeouts / circuit breaking** — что делать, если B тормозит или лежит.
- **Load balancing** — раскидывать вызовы между копиями B.
- **Traffic shaping** — лить N% на canary-версию B (вот тут mesh и canary пересекаются).
- **Observability** — трассировка каждого вызова сквозь систему.

### Что такое mesh

Рядом с каждым сервисом ставится **sidecar-прокси** (Istio/Envoy, Linkerd, Consul Connect). Весь межсервисный трафик идёт через эти sidecar'ы, которые прозрачно делают всё перечисленное. Сверху — control plane, который их конфигурирует.

### Цена

Sidecar-прокси **на каждый сервис** (память + латентность на каждый хоп) + отдельный control plane. На нодах по 1–2 ГБ это абсурдный оверкилл — sidecar'ы съедят больше, чем сами сервисы.

### Насколько это нужно тебе

**Не нужно вообще**, и причина структурная: у тебя топология — **звезда**, а не сетка. Каждый сервис общается с PetBox (центр) и с внешними API, но **не друг с другом**. Нет east-west трафика → нет проблемы, которую решает mesh.

День, когда сервисам понадобится звать друг друга, **не** наступит «нужен mesh». Дешёвый ответ для твоего масштаба:

- регистрировать эндпоинты сервисов в PetBox (у тебя уже есть config!) и резолвить друг друга через PetBox;
- либо один общий shared reverse-proxy с понятными именами.

Mesh — это тяжёлый промышленный ответ на проблему сотен взаимосвязанных сервисов, которой у тебя нет и в обозримом будущем не предвидится.

**Вывод:** mesh — не про тебя на уровне топологии, а не масштаба. Даже вырастешь в числе — звезда вокруг PetBox остаётся, и mesh не понадобится.

## Свод: что из этого закладывать

| Механизм | Нужен сейчас? | Дешёвая версия в нашей модели |
|---|---|---|
| Rolling | Нет (боты без трафика; дорого по RAM) | shared reverse-proxy для редких HTTP-сервисов, точечно |
| Canary | Нет (полный split не нужен) | **canary by node** — staging `image_digest` на одной ноде, ручной промоут |
| Service mesh | Нет (топология «звезда», нет east-west) | эндпоинты в config / один shared-proxy, если вдруг |

Триггер «пора смотреть на Nomad/настоящий оркестратор» остаётся **один**: bin-packing под давлением (много сервисов, мало RAM, надо умно раскладывать с оптимизацией). Failover для stateless-ботов и перечисленное выше к этому триггеру **не** относятся — они либо не нужны, либо берутся дёшево.

## Что реально набирает актуальность (и почти бесплатно)

Из твоей же формулировки — «деплой, мониторинг и управление». Заметь, что **мониторинг** частично получается даром из той же модели:

- канал **heartbeat/status** от агента уже отвечает на «что где запущено и здорово ли» — это базовый мониторинг без отдельной системы;
- **логи** уже уходят в PetBox (есть log-модуль) — централизованный просмотр есть.

То есть основная ценность ближнего горизонта — deploy + start/stop/move + базовый мониторинг — закрывается control-plane моделью без rolling/canary/mesh.

## Открытые вопросы для следующего захода

- Язык агента: Python-MVP (дёшево поднять на Ubuntu) vs C# AOT single-file (шарит контракт `poll`/`heartbeat` DTO с PetBox, нет дрейфа схемы).
- Модель данных: `node(id, tags, last_seen, ephemeral)` + `deployment(service, node, image_digest, key_ref, desired, relocatable, config_hash)`.
- Контракт agent API: `GET /agent/poll?node=…` + `POST /agent/heartbeat`, авторизация скоупленным ключом ноды.
- Правило ресчедулинга: N пропущенных heartbeat → выбрать ноду по тег-вектору+RAM → переставить `node` → старый агент при возврате видит, что deployment не его, и самогасится (дешёвый фенсинг; режим отказа — двойной запуск, не потеря данных; только для `relocatable:true`).
- CI после внедрения: репо проекта только собирает+пушит образ и сообщает PetBox новый digest; деплоит агент. SSH из CI и хост-секреты в репах исчезают.
