## Извлечение — шов

**Новая сборка `PetBox.Tasks.Engine` (чистая, ноль linq2db):**
- переносим уже-чистые: `WorkflowEngine`, `MethodologyRuntime`, `MethodologyDefinition`,
  `MethodologyPresets`, `MethodologyGuide`, `Workflow.cs`;
  [после Этапа 0: слайс 1 увёз восемь уже-чистых файлов, но `MethodologyGuide` в них НЕ попал —
  он сидел рядом с linq2db-`PlanNode` и уехал отдельным слайсом 2 вместе с чистыми view-типами
  (`MethodologyGuideView`/`MethodologyInvariant`), которых этот список не предвидел]
- **новый `GuardEngine`** (~200 строк) — чистые версии `RequireDefinitionLinks`,
  `ValidateLinkTargets`, `RequireBlockers`, `RequirePreconditionArtifacts`. Каждый guard режется на
  (а) выборку — остаётся в сервисе, (б) чистое суждение — уезжает в движок.

**`TasksService` становится IO-границей:** предвыбрать всё нужное ОДИН раз → отдать движку →
получить решение → записать + применить эффекты. Остаётся: board-CRUD, чтение/поиск, стейджинг
выборок, применение эффектов (`TaskTransitionEffects` — остаётся DB-bound), **минимальная**
валидация ввода. Основу валидации даёт движок.

**Вход/выход движка (форма — уточняется реализацией):**

> **[после Этапа 0] Эскиз разошёлся с реализацией в одном месте — сознательно.**
> `desiredStates`/`priorStates` в контекст НЕ уехали: они **аргументы `GuardEngine.Decide`**, а не
> поля контекста. Причина: контекст предвыбирается ОДИН раз и переигрывается retry-циклом без
> пересборки, а состояния меняются КАЖДЫЙ проход — держать их в «выбранном однажды» объекте
> значит сделать слово «однажды» ложью. Остальная форма приземлилась как нарисована.
> Реальная сигнатура: `Decide(ctx, desired, prior, specRefsRaw, blockedByRaw, ideaRefsRaw)`.

```
MethodologyEngineContext(          // всё предвыбрано сервисом, БД внутри движка НЕТ
    runtime, kindSlug,
    desiredStates, priorStates,             // проекции NodeState, не linq2db-PlanNode (усл.4)
                                            // [после Этапа 0: АРГУМЕНТЫ Decide, не поля контекста]
    // СЫРЫЕ кандидаты резолюции (движок резолвит сам, усл.3), не готовые NodeId:
    instanceIdeaBoards, specNodes, boardPriorDesired,
    nodeIndex,                              // активные узлы проекта — один раз
    blockerEdgesByNodeId,                   // активные blocks-рёбра в узлы (усл.2)
    partOfChildrenByNodeId,                 // active part_of дети (delete-guard, усл.2)
    commentTagsByNodeId )                   // теги коммов для precondition — предвыбраны
→ MethodologyEngineDecision(               // ТОЛЬКО пре-write вердикты (усл.1)
    workflowValidation, resolvedRefs,
    linkViolations, blockerViolations, artifactViolations )
    // effectsToApply НЕТ: эффекты остаются императивными post-write с live-чтениями,
    // по `landed`-подмножеству (усл.1)
```

> **[после Этапа 0] Выход приземлился в другой форме — и это важно для Этапа 1.**
> Вместо по-guard'ных вёдер виолаций (`linkViolations`/`blockerViolations`/`artifactViolations`)
> реализация несёт `MethodologyEngineDecision(Verdicts, SpecRefs, BlockedBy, IdeaRefs)`, где
> `Verdicts` — **fail-fast**: движок гоняет 7 стадий в историческом порядке вызова и стопается на
> ПЕРВОМ отказе, ровно там, где старый код бросал исключение. Сегодня в списке максимум одна
> запись. Это не временное упрощение: пройти дальше отказа значит судить по полурезолвленной
> ref-карте и ПРОИЗВОДИТЬ вердикты, которых никогда не было (провалившийся `ResolveSpecRefs` не
> кладёт ключ в `specRefs` → `RequireDefinitionLinks` немедленно «обнаружит» отсутствующий specRef
> и обвинит тот же узел ложным сообщением). Форма списка оставлена, чтобы поздний слайс мог
> расширить её **по-guard'но и со своим паритет-доказательством** — не «заодно».
> `workflowValidation` в `Decision` не поехал: `ApplyWorkflow` остался в сервисе (он строит
> `PlanNode`, а не судит), движок зовут ПОСЛЕ него. Вердикт несёт `VerdictKind`
> (`InvalidArgument`→`ArgumentException` / `InvalidOperation`→`InvalidOperationException`), потому
> что тип исключения нагружен смыслом (`Assert.Throws<T>` матчит точно) — чистый движок обязан его
> НАЗВАТЬ, иначе сервис угадывает. Склейка с exception-швом — ровно одна точка,
> `TasksService.ToRefusal`.

**Два зайца:** предвыбор-в-контекст-однажды заодно **убивает гипотезу B** — дублирующие
`BuildNodeIndex`/`boards.ListAsync` схлопываются в один проход. Тестируемость + перф одним движением.

**Самые сросшиеся места (и как инвертировать):**
1. **Precondition читает комменты в середине решения** (`RequirePreconditionArtifactsAsync`). → движок
   ДЕКЛАРИРУЕТ «нужны теги коммов для узла X», IO-слой предвыбирает все комменты доски один раз и
   отдаёт `commentTagsByNodeId`. [после Этапа 0: декларация — `GuardEngine.NeedsCommentTags`,
   предвыборка — `TasksService.BuildEngineContextAsync`]
2. **Резолв `ideaRef` лезет в чужие доски** (`ResolveIdeaRefsAsync`). → сервис предвыбирает
   idea-доски инстанса, отдаёт map; движок не ходит в БД. [после Этапа 0: `GuardEngine.ResolveIdeaRefs`
   резолвит по сырым `ctx.Boards`]
3. **`BuildNodeIndexAsync` зовётся дважды в пути.** → вычислить раз в `UpsertAsync`, прокинуть во все
   решатели. [после Этапа 0: один раз в `BuildEngineContextAsync` → `ctx.NodeIndex`]

## Почему ЭТАП 0, а не попутно

Редизайн (утилиты `01-03`) тяжело правит именно эти гайарды и резолверы. Делать его на god-class'е —
дорого и рискованно (прод-инцидент с DefaultView уже показал, чем оборачивается размытая логика).
Порядок: **(0) извлечь движок под чистые тесты → (1) редизайн на нём дешёвыми правками → (2)
помеченные открытые вопросы `03`**. Извлечение — behavior-preserving рефактор, приземляется с
бит-в-бит паритет-тестами; та же pure-сюита потом стережёт редизайн.

## Условия старта (Fable: GO WITH CONDITIONS, сверено с кодом)

Вердикт положительный; пять условий — контрактные уточнения, ни одно не рушит замысел. С усл.1-3
контракт движка (`runtime + состояния + сырые предвыборки → вердикты`) стабилен относительно ВСЕХ
новых примитивов редизайна (`blocksGate`, `Effect.onLeave`, `requiredArtifacts`, declared link-kinds,
resolve-by-writing-end) — пере-извлечения на Этапе 1 не потребуется.

1. **`Decision` несёт только пре-write вердикты; эффекты — НЕ решать заранее.** `CloseBlocksOnLeaveAsync`
   читает live-relations ПОСЛЕ записи blocks-ребра (`LinkRefsAsync`) в том же вызове;
   `TaskTransitionEffects.RunTransitionEffectsAsync` мутирует и перечитывает `stillBlocked`
   (release последнего блокера); пост-write работает по `landed` (вычисляется в `UpsertAsync` из
   `r.Conflicts`, т.е. определяется конфликтами `TemporalStore`, непознаваемыми пре-write). Эффекты
   остаются императивными post-write с live-чтениями. `effectsToApply` убран.
   [после Этапа 0: выполнено; `effectsToApply` не заводился. **Это же условие вычёркивает обещание
   pure-тестов на эффекты из «Выигрыша по тестам» выше.**]
2. **В контекст добавить рёберные данные.** `RequireBlockersAsync` уже сегодня нужны активные
   blocks-рёбра в узел, delete-guard — active part_of дети (`ActivePartOfChildrenAsync`); `blocksGate`
   из редизайна — то же. Без `blockerEdgesByNodeId`/`partOfChildrenByNodeId` не воспроизвести даже
   текущее. [после Этапа 0: выполнено, `ActivePartOfChildrenAsync` удалён —
   `ctx.PartOfChildrenByNodeId` его заменил]
3. **Резолюцию specRef/ideaRef/blockedBy втянуть В движок.** Её ошибки (`ResolveIdeaRefsAsync`,
   `ResolveSpecRefs`, `ResolveBlockedBy`) — полноправные вердикты с `ForNode`
   в partial-каскаде retry-цикла. Оставить IO-side = кусок решателя без чистых тестов, и Этап 1
   (`resolve-by-writing-end`) всё равно сдвинул бы границу. Контекст несёт СЫРЫХ кандидатов, движок
   резолвит чисто. [после Этапа 0: выполнено — `GuardEngine.ResolveSpecRefs`/`ResolveIdeaRefs`/
   `ResolveBlockedBy`, стадии 1-3 `Decide`]
4. **«Ноль linq2db» невозможен с текущим `PlanNode`** (`Data\PlanNode.cs` — `using LinqToDB.Mapping`).
   Решение до старта: **лёгкая проекция `NodeState`-record в Engine + маппинг в сервисе** (рекомендую —
   ниже риск), либо перевод `PlanNode` на fluent-mapping и вынос в чистую сборку. Кусок работы, учесть
   в объёме. [после Этапа 0: взят первый вариант — `NodeState` (Key, PrevKey, NodeId, Status, Type);
   `PlanNode` границу Engine не пересекает. `DeliveryEngine` переиспользовал `NodeState` без новых
   полей — роллап читает строгое подмножество (NodeId/Status/Type)]
5. **Стандарт паритета — не «бит-в-бит».** NodeId — свежий Guid на каждый проход retry-цикла
   (`ApplyWorkflow`), буквальное равенство недостижимо и не нужно. Реальный стандарт: **вся существующая сюита зелёная**
   (`tests\PetBox.Tests\Tasks\` — 386 `[Fact]/[Theory]`/43 файла, прямо по upsert-пути: `QuartetTests`,
   `TasksMethodologyWorkFsmTests`, `BatchPartialApplyTests`, `TasksMethodologyRefsTests`,
   `WorkflowEngineTests`…) **+ равенство error-строк** в новых unit'ах; retry-цикл partial-режима
   сохранить структурно (движок переигрывается по стабильному контексту), не заменять одно-проходным
   сбором всех виолаций.

## Оговорка честности

Проверено фактом: 2348 строк, чистота `WorkflowEngine`/`MethodologyRuntime` (0 DB-зависимостей),
25 мест board-выборки, 4 guard'а вне движка, `TaskTransitionEffects` DB-bound. Проценты разбивки,
точный по-методный раскрой и «~150 тестов» — оценка первичной разведки (её делала Haiku), уточняется
при реализации. Форма `MethodologyEngineContext/Decision` — эскиз шва, не финальный контракт.

**[после Этапа 0] Где эскиз разошёлся с реализацией** (все три — сознательно, ни одно не рушит
замысел): `desiredStates`/`priorStates` стали аргументами `Decide`, а не полями контекста; `Decision`
приземлился fail-fast списком вердиктов вместо по-guard'ных вёдер; `MethodologyGuide` уехал не
слайсом 1, а слайсом 2, вместе с чистыми view-типами. Обещание pure-тестов на эффекты снято как
противоречащее усл.1; singleton назван новым слайсом, а не недоделкой; delivery-роллап доставлен
поздним слайсом поверх готового контекста.

**Самое хрупкое место наследства — предикаты `needs*` в `BuildEngineContextAsync`.** Их корректность
держится на том, что контекст строится из `nodes` (патчи КАК ПОДАНЫ) и `prior` — единственных данных,
которые retry-цикл НЕ сужает. Правило: сужать можно СОДЕРЖИМОЕ выборки (меньше id, меньше колонок),
но НИКОГДА множество патчей, из которых она выведена. Сегодня инвариант **латентный**: `live` и
`nodes` наблюдаемо расходятся только на delete-пути (закреплено `EngineContextStabilityTests`).
Выстрелит он, когда в контекст добавят поле, зависящее от батча КРОСС-УЗЛОВО — например, индекс,
суженный до referenced-ids: тогда выпавший узел утащит данные выжившего, и поломка будет молчаливой.
Этап 1 — это ровно тот, кто такое поле добавит.
